## 发现

当前实现已经不再表现出此前的左右反向或上下反向类问题。剩余可见差异主要来自渲染模型本身：

- 当前效果仍是较大的屏幕空间四边形或梯形；
- 光片内容来自实时视频采样，再叠加色彩和网格样式；
- 背景标记或人脸会在光片中重复出现，这是当前实时采样 shader 的设计结果；
- 光片虽然能产生视觉旋转，但不会成为折叠的多面物体。

参考视频的目标不同：

- 手部/指尖锚定的空间模板；
- 多个面具有独立材质；
- 具有薄片厚度感和边缘高光；
- 具有透视缩放和近大远小；
- 会进行面片翻转、折叠和带状旋转；
- 运动感觉绑定在手指锚点上，而不是只绑定在摄像头预览中的二维平面上。

因此，继续修正二维光片偏移，大概率只是在打磨一个过渡 renderer。它可以让当前 demo 更稳定，但不会收敛到参考视频的目标行为。

## 决策

下一步应进入三维模板实现。

当前二维 renderer 只保留为坐标空间和实时视频采样的校准/调试工具，不再把“让扁平光片继续接近参考视频”作为主要迭代方向。

仍然有价值的二维工作应限定为诊断性工作，例如增加一个无色彩、无网格的 raw sampling 调试样式，用于确认 display-space 到 video-uv 的映射仍然正确。

## 技术债复核

按 brooks-debt 视角，本次主要风险是 **Domain Model Distortion** 和 **Change Propagation**：如果继续在 `light-sheet-renderer` 内实现折叠、多面、指尖锚定行为，就会强迫当前扁平光片模型表达另一个领域概念。

补救方式：按 ADR-0002 新建 spatial-template 边界，并让 `light-sheet-renderer` 保持为过渡 renderer。这样后续新增模板、折叠、材质或锚点时，不会每次都横跨 gesture、sampling、renderer 和 UI 多处重写。

## 推荐的下一步实现

实现 ADR-0002 的第一个里程碑：

- 新增经过测试的手势锚定框架，由手部 landmarks 推导；
- 新增空间模板模型，包含顶点、面、材质 id 和折叠状态；
- 渲染一个最小 Three.js 带状/薄棱柱模板，支持多个纹理面；
- 将实时视频采样保留为可选材质来源，而不是唯一效果模型；
- 使用指尖/手部锚点驱动位置、旋转、缩放和折叠；
- 遮挡和人体分割放到后续阶段，先稳定三维模板运动。

## 下一次变更验证方案

自动化：

- 手部 landmark 到 anchor-frame 转换的单元测试；
- mesh 构造、面顺序、折叠状态的单元测试；
- 手势状态转换测试：无手、单手、双手；
- renderer smoke 测试，确认所有模板材质无 shader 编译错误；
- 完整 `npm test`、`npm run build`、中英文文档配对检查和 `git diff --check`。

真实设备：

- 只有一只手可见时，只有在明确选择支持单手锚点的三维模板模式下才应渲染；
- 两只手可见时，应生成空间模板，而不是扁平屏幕空间光片；
- 手指左/右/上/下移动时，模板应沿可见同方向移动；
- 手指前后移动时，应体现缩放/透视变化，而不是只做平移；
- 旋转或折叠过程中，模板至少应出现两个视觉上不同的面；
- 新录屏需与 `reference_contact_1fps.jpg` 和 `reference_segment_000_024_4fps.jpg` 对比。
