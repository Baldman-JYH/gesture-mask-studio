## 可见差距

当前部署效果仍然更像一个贴在屏幕空间里的半透明薄片。它可以跟随手部锚点移动，但还没有稳定呈现为真正的三维模板模型。

参考视频的目标明显不同：

- 展示的是手部锚定的三维模板，不是单个三角形；
- 运动过程中可以看到多个面；
- 不同面具有不同视觉处理，而不是统一的透明染色；
- 折叠、带状翻转和旋转会显露侧面、厚度和透视；
- 运动感觉绑定在指尖锚点上，位置、旋转和折叠随手势变化。

当前实现有两个直接缺陷：

- 一个物理手可能被计为两只手，因为业务层直接使用了 MediaPipe 原始检测结果，没有做重复手过滤；
- 当前程序化 mesh 仍然过于接近 display-space 内的扁平薄棱柱，因此视觉上不像折叠的多面体三维模型。

## 架构评估

当前技术栈仍然可行：

- MediaPipe Hands 能在浏览器内实时提供手部 landmarks；
- Three.js 能渲染程序化三维 mesh、纹理面、厚度和透视；
- GitHub Pages 静态部署仍然适合个人开发者，因为摄像头在 HTTPS 下可用，效果完全在客户端运行。

问题不是技术栈选择错误，而是当前实现边界还不够准确。项目应保留现有架构，但强化领域边界：

- gesture tracking 输出经过过滤且稳定的手部锚点帧；
- spatial-template-model 负责 mesh 拓扑、折叠状态、材质 id 和模板变体；
- spatial-template-renderer 负责把 mesh 映射为 Three.js buffer 和材质；
- camera 与 UI 层消费过滤后的手势/模板状态，而不是直接消费原始检测数量。

## 决策

继续使用当前 browser + MediaPipe + Three.js 技术栈。

不要切换到服务端渲染、NVIDIA 专用运行时或静态图片生成方案。下一轮优化应把模型从薄棱柱改为折叠多面体三维模板，并在 UI 与 renderer 派生状态前过滤重复手部检测。

## 下一步必须修正

- 对共享同一物理锚点的手部检测做重复过滤。
- 顶部状态手数来自过滤后的 gesture anchor frame，而不是原始 detector 数量。
- 将单手三角形/楔形替换为小型折叠矩形模板。
- 将双手扁平 ribbon prism 替换为非共面的折叠模板 mesh，包含实时画面、次级面、背面、强调面和边缘材质组。
- 保留实时视频采样作为其中一个面材质，而不是整个效果模型。

## 验证重点

自动化验证必须证明：

- 重复检测会收敛为一只可用手；
- 分离的真实双手仍然产生双手帧；
- 单手 mesh 不再是三角形；
- 双手 mesh 至少包含多个可见面类型；
- 生成的顶点具备足够非共面性，可以渲染为三维模型；
- 材质 id 能映射到不同 renderer group。

真实设备验证必须确认：

- 一个物理手不再显示 `2 hands`；
- 单手模式显示紧凑的折叠三维模板，而不是三角形；
- 双手模式在移动中能看到多个面和边缘厚度；
- 左右上下移动仍与摄像头预览方向一致；
- 前后远近移动会产生透视变化，而不是只平移一张扁平薄片。
