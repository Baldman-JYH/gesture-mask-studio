# Brooks 技术债架构评审

English version: [brooks-debt-architecture-review.md](brooks-debt-architecture-review.md)

## 评审目标

本评审用于识别项目早期架构中可能导致后期重构的技术债风险，重点关注：

- 认知负担；
- 变更传播；
- 知识重复；
- 偶然复杂度；
- 依赖混乱；
- 领域模型失真。

## 评审范围

- 视频效果分析。
- 产品需求和业务逻辑。
- 技术架构。
- 运行时契约。
- GitHub Pages 部署方案。
- MVP 实现计划。

## 结论

当前架构没有发现阻塞性技术债。核心概念“手势驱动实时采样光片”足够明确，模块边界合理，适合继续实现和迭代。

## 关键判断

### 认知负担

风险：如果继续使用“蒙版”解释核心效果，容易误导实现为静态遮挡。

处理：文档中明确修正为“实时采样光片”，`mask` 仅保留在项目名中。

### 变更传播

风险：如果样式、手势识别和渲染耦合，新增一种纹理会迫使修改多个模块。

处理：通过 `LightSheetStylePreset` 隔离样式扩展，手势引擎和渲染器只消费标准契约。

### 知识重复

风险：视频采样、光片几何、样式预设如果在多个组件中重复表达，会产生漂移。

处理：集中定义运行时类型和模块职责。

### 偶然复杂度

风险：过早引入后端、云 GPU、复杂编辑器或账号系统会超过 MVP 需求。

处理：首版限制为纯前端静态应用，优先浏览器本地能力。

### 依赖方向

风险：领域逻辑依赖 MediaPipe 或 Three.js 会导致供应商替换困难。

处理：MediaPipe 限制在 `hand-tracking`，Three.js 限制在 `light-sheet-renderer`。

### 领域模型

风险：将效果理解成“人脸特效”会偏离用户需求。

处理：领域模型明确为“光片背后的所有实时画面都被采样和重新渲染”。

## 维护建议

- 新增功能前先判断属于哪个模块边界。
- 新增样式时先扩展 preset。
- 新增识别模型时保持 `TrackedHand` 输出不变。
- 新增渲染效果时保持 `LightSheetRenderInput` 不变。
- 每次架构相关变更都应同步更新中英文文档。
