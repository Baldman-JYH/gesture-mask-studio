# ADR-0001：采用手势驱动实时采样光片

English version: [adr-0001-realtime-scene-sampling-light-sheet.md](adr-0001-realtime-scene-sampling-light-sheet.md)

## 状态

已接受。

## 背景

参考视频中的效果并不是简单“蒙版遮挡”，也不是只渲染人脸。光片内部会显示其背后的实时摄像头内容，并叠加不同视觉样式。用户强调基础需求是实时动态渲染，且后续要能扩展新的样式和功能，避免重构。

## 决策

采用“手势驱动实时采样光片”作为核心架构模型：

- 手部关键点驱动光片几何。
- 当前摄像头帧作为 WebGL video texture。
- 光片内部根据屏幕几何实时采样摄像头内容。
- 样式通过 `LightSheetStylePreset` 配置扩展。
- 渲染由 WebGL/Three.js 完成。

`mask` 只作为项目名中的习惯用词，不作为核心技术概念。

## 被拒绝的方案

### 静态图片覆盖

拒绝原因：

- 无法显示光片背后实时变化的画面。
- 不能满足用户明确要求的实时动态效果。
- 后续新增样式会变成贴图堆叠，缺乏真实采样能力。

### 只做人脸渲染

拒绝原因：

- 用户已明确说明人脸只是示例。
- 光片背后的所有画面都应该被实时采样，包括手、衣服和背景。

### 后端/云端视频处理

拒绝原因：

- 增加部署和隐私成本。
- GitHub Pages 无法承载后端。
- 当前 MVP 可在浏览器本地完成。

## 结果

正面影响：

- 与参考视频核心行为一致。
- 可部署为静态站点。
- 后续样式扩展成本低。
- 摄像头帧不上传，隐私边界清晰。

代价：

- 需要 WebGL 和 WebAssembly 支持。
- 低端设备可能需要降级策略。
- 手势识别和渲染性能需要持续验证。

## 架构约束

- 手势引擎必须保持纯逻辑，不依赖 MediaPipe 或 Three.js。
- MediaPipe 只能出现在 `hand-tracking` 适配层。
- Three.js/WebGL 只能出现在渲染模块。
- 样式新增必须优先通过 preset 完成。
- 验证必须包含真实摄像头和实时采样检查。
